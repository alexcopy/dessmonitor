POLLING AND CONCURRENCY
- Recursive setTimeout scheduling (no setInterval)
- Maximum one fetch in flight (AbortController)
- 10-second request timeout via AbortController
- 5-second normal visible polling interval
- 10-second offline polling interval
- 30-second hidden-tab polling interval
- Bounded backoff: consecutiveFailures 0-2 -> normal, 3+ -> offline
- Immediate poll on visibility restore (becoming visible)
- beforeunload + pagehide cleanup: abort controller, clear timers
- HTTP 401 -> clear all state, navigate to /login
- HTTP 4xx -> do not retry aggressively, mark degraded if online
- HTTP 5xx -> bounded retry, mark offline after threshold
- Network failure -> bounded retry, mark offline after threshold
- AbortError -> silently skipped (not treated as failure)
- One-at-a-time: next poll scheduled only after current settles

CONNECTION STATES
connecting — page load, no response yet
online     — successful response within stale threshold (15s)
stale      — 15s without fresh response
degraded   — API status is "degraded" or "blocked"
offline    — 60s without communication
