APPROVED CANONICAL ARTIFACT CONTRACT

All agent workflow artifacts from PR 0034a onward must use these exact
paths and names:

  plan            -> .project-memory/pr/<pr-id>/PLAN.txt
  plan-review     -> .project-memory/pr/<pr-id>/reviews/PLAN_REVIEW.txt
  coder           -> .project-memory/pr/<pr-id>/CODER_REPORT.txt
  precommit-review -> .project-memory/pr/<pr-id>/reviews/PRECOMMIT_REVIEW.txt

Canonical format rules:

  1. All files use plain UTF-8 text encoding.
  2. No YAML, JSON, XML, or Markdown document structure is required
     or expected.
  3. Files use readable headings and sections separated by blank lines.
  4. Files may contain ordinary prose, labels, lists, paths, and
     command results formatted as plain text.
  5. Files must accurately report commands that were actually run.
  6. Files must not contain secrets, credentials, tokens, or keys.
  7. Filenames use exact casing: PLAN.txt, PLAN_REVIEW.txt,
     CODER_REPORT.txt, PRECOMMIT_REVIEW.txt.
  8. PLAN.txt and reviews/PLAN_REVIEW.txt are locked after the
     planning commit. Coder and precommit-review agents may not
     modify them.
  9. Review artifacts are always inside a reviews/ subdirectory
     relative to the PR directory.
  10. CODER_REPORT.txt is the implementation agent's output.
  11. PRECOMMIT_REVIEW.txt is the precommit-review agent's output.
  12. Any agent generating any artifact must not stage, commit, push,
      or create a Pull Request unless the task prompt explicitly
      assigns that action.
  13. An artifact that reports commands not actually run, or that
      fabricates validation results, is invalid.

TRANSITION BOUNDARY

The transition boundary uses STRUCTURAL NUMERIC IDENTIFIER PARSING,
not raw ASCII lexical comparison of the full directory name.

IDENTIFIER ALGORITHM

Parse the PR directory basename into:

  numeric_part: the leading decimal digits (0-9) before the first
    non-digit character, or empty string if the basename does not
    start with a digit.

  suffix_part: the lowercase alphabetic character immediately after
    the numeric_part, before the first hyphen, or empty string if
    no alphabetic suffix exists.

Boundary constant: numeric_part = "0034", suffix_part = "a"

Classification procedure:

  1. If numeric_part is empty: the directory is NON-NUMERIC.
     Check against the frozen historical allowlist (see section
     NON-NUMERIC HISTORICAL ALLOWLIST).
     - If allowlisted: legacy-accepting. Accept all artifacts.
     - If not allowlisted: REJECTED. Unknown non-numeric directory
       name. Legacy names are not permitted for new directories.

  2. If numeric_part is not empty: the directory is NUMERIC.
     Compare numerically:
     - If numeric_part < 34: LEGACY. Accept all artifacts.
     - If numeric_part > 34: CANONICAL. Require canonical format.
       Reject legacy format.
     - If numeric_part == 34: compare suffix_part:
       - If suffix_part is empty: LEGACY.
       - If suffix_part < "a": LEGACY.
       - If suffix_part == "a": CANONICAL (boundary).
       - If suffix_part > "a": CANONICAL.

Note: numeric_part is parsed as an integer for numeric comparison.
The leading zeros (e.g., "0034") are ignored in numeric comparison.
34 equals 0034 as integers.

Required examples:

  0034 -> numeric_part=34, suffix=empty, 34==34, suffix empty < "a" -> LEGACY
  0034a -> numeric_part=34, suffix="a", 34==34, suffix=="a" -> CANONICAL
  0034b -> numeric_part=34, suffix="b", 34==34, suffix>"a" -> CANONICAL
  0035 -> numeric_part=35, suffix=empty, 35>34 -> CANONICAL
  0040 -> numeric_part=40, suffix=empty, 40>34 -> CANONICAL
  0018a -> numeric_part=18, suffix="a", 18<34 -> LEGACY
  0028b -> numeric_part=28, suffix="b", 28<34 -> LEGACY
  0001 -> numeric_part=1, suffix=empty, 1<34 -> LEGACY
  fix-auth-validation-python-discovery -> numeric_part=empty, allowlisted -> LEGACY
  unknown-non-numeric-future -> numeric_part=empty, not allowlisted -> REJECTED

LEGACY STAGE MAPPING

For pre-boundary directories, stage detection uses this exact mapping.
The validator checks artifacts in this order:

  COMPLETE stage (legacy):
    PRECOMMIT_REVIEW.yaml exists in root directory.
    OR reviews/precommit-review.yml exists.

  IMPLEMENTATION stage (legacy):
    CODER_REPORT.txt exists.
    PRECOMMIT_REVIEW.yaml or reviews/precommit-review.yml does NOT
    exist.

  PLANNING stage (legacy):
    PLAN_REVIEW.yaml exists in root directory.
    OR reviews/plan-review.yml exists.
    CODER_REPORT.txt does NOT exist.

  DRAFT stage (legacy):
    PLAN.md exists.
    No PLAN_REVIEW.yaml, reviews/plan-review.yml, CODER_REPORT.txt,
    PRECOMMIT_REVIEW.yaml, or reviews/precommit-review.yml exists.

  EMPTY:
    No artifacts at all. PASS with informational message.

Note: The validator does NOT require all legacy directories to have
the same layout. Each directory is validated against its actual
present artifacts. The stage detection merely determines which checks
are appropriate.

A legacy directory with PRECOMMIT_REVIEW.yaml but no PLAN_REVIEW.yaml
or PLAN.md (e.g., PR 0007) is handled correctly: it is classified as
LEGACY (pre-boundary), detected as COMPLETE (PRECOMMIT exists), and
no canonical-format checks are applied.

A legacy directory with only CODER_REPORT.txt (none exist in the
repository, but possible) would be detected as IMPLEMENTATION and
accepted because no canonical-format checks apply.

CANONICAL STAGE MAPPING

For canonical (post-boundary) directories, stage detection uses the
deterministic existence order:

  COMPLETE stage:
    reviews/PRECOMMIT_REVIEW.txt exists.
    Validation: all four canonical artifacts must exist and be valid.

  IMPLEMENTATION (coder complete) stage:
    CODER_REPORT.txt exists.
    reviews/PRECOMMIT_REVIEW.txt does NOT exist.
    Validation: PLAN.txt, reviews/PLAN_REVIEW.txt, and
    CODER_REPORT.txt exist and are valid.

  PLANNING COMPLETE stage:
    reviews/PLAN_REVIEW.txt exists.
    CODER_REPORT.txt does NOT exist.
    Validation: PLAN.txt and reviews/PLAN_REVIEW.txt exist and are
    valid.

  PLANNING DRAFT stage:
    PLAN.txt exists.
    reviews/PLAN_REVIEW.txt does NOT exist.
    CODER_REPORT.txt does NOT exist.
    Validation: PLAN.txt exists and is valid. Future-stage artifacts
    are NOT required.

  EMPTY directory:
    No artifacts.
    PASS with informational message. No format validation applied.

  REJECTION conditions for canonical directories:
    - PLAN.md exists -> REJECT (legacy PLAN name).
    - PLAN_REVIEW.yaml or reviews/plan-review.yml exists
      -> REJECT (legacy review name).
    - PRECOMMIT_REVIEW.yaml or reviews/precommit-review.yml exists
      -> REJECT (legacy review name).
    - Any .yaml or .yml file inside the PR directory or reviews/
      subdirectory -> REJECT (legacy format extension).
    - PLAN.txt with wrong casing -> REJECT.
    - PLAN_REVIEW.txt not inside reviews/ -> REJECT.
    - PRECOMMIT_REVIEW.txt not inside reviews/ -> REJECT.
    - Non-numeric directory NOT in frozen allowlist -> REJECT.

  The validator must not require future-stage artifacts prematurely.
  A directory at PLANNING DRAFT stage passes even though
  CODER_REPORT.txt and reviews/PRECOMMIT_REVIEW.txt are absent.

VALIDATOR DESIGN

The updated scripts/check-project-memory.sh implements:

  1. Keep existing governance file checks unchanged (AGENT_STANDARD.txt
     etc.).

  2. Replace the hardcoded PR 0003 artifact check with generic
     iteration over all PR directories.

  3. For each PR directory under .project-memory/pr/:

     a. Extract basename.
     b. Classify using the STRUCTURAL IDENTIFIER ALGORITHM:
        - Parse numeric_part and suffix_part.
        - If numeric_part is empty: check frozen allowlist.
          Allowlisted -> legacy, not allowlisted -> reject.
        - If numeric_part < 34: legacy.
        - If numeric_part > 34: canonical.
        - If numeric_part == 34 and suffix_part is empty: legacy.
        - If numeric_part == 34 and suffix_part == "a": canonical.
        - If numeric_part == 34 and suffix_part > "a": canonical.

     c. Detect stage using artifact existence:
        - Legacy directories use LEGACY STAGE MAPPING.
        - Canonical directories use CANONICAL STAGE MAPPING.

     d. Validate artifacts:
        - Legacy: accept any layout. No format checks.
          Report ambiguous duplicate stage artifacts if both legacy
          and canonical artifacts for the same stage exist.
        - Canonical: require exact filenames, reject legacy names,
          reject non-allowlisted non-numeric directories, reject
          wrong casing, reject wrong reviews placement.

  4. Accumulate errors. Exit 0 only if zero errors.

The validator works without network access, Git history, origin/master
remote, or GitHub CLI. It uses only local file existence tests and
string operations. It does not mutate any file.

RISKS AND MITIGATIONS

Risk: Structural identifier parsing is more complex than string
  comparison.
  Mitigation: The parser extracts only leading digits and single
  optional alphabetic suffix. Pure shell string operations suffice.
  No regex library needed. The added complexity handles the known
  non-numeric historical directory correctly.

Risk: New non-numeric PR directory rejected by validation.
  Mitigation: This is intended behavior. New directories must use
  the numeric prefix convention. If a genuine non-numeric directory
  is required, the allowlist must be amended by a separate plan.

Risk: Legacy stage detection misses a variant.
  Mitigation: The inventory covers every observed variant in the
  repository. The detection uses flexible ordering (check existence
  of each artifact type). Any additional variant discovered later
  can be added to the inventory during implementation.
