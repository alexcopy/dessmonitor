WARNINGS

1. Latent bug in parse_pr_id: all-digit basename without slug falsely
   classified as non-numeric.

   The check `if [ "$numeric_part" = "$basename" ]; then numeric_part=""; fi`
   treats a bare all-digit basename (e.g., "0035" with no slug) as having no
   leading digits, falling through to the non-numeric allowlist check. This
   does NOT affect any current repository directory because every numeric PR
   directory has a slug suffix (e.g., "0035-runtime-heartbeat-..."). The bug
   is latent and should be corrected by changing the check to verify that
   numeric_part is empty rather than comparing the full basename. Not a
   current blocker, but should be addressed before creating any bare-digit
   fixture tests or future uses.

2. The parse_pr_id function has redundant check. If numeric_part is empty
   (extracted as ""), the `if [ "$numeric_part" = "$basename" ]` check is
   always false (empty string never equals a non-empty basename), so the
   early-return path for "no leading digits" actually only triggers for
   all-digit basenames. The function still works correctly because classify
   handles empty numeric_part at its own level. This is cosmetic but could
   be confusing to maintainers.

3. No separate fixture-based test harness was created. The CODER_REPORT
   notes that the real repository directories serve as the test. While this
   provides comprehensive coverage today, it means the validator's
   rejection behavior (unknown non-numeric, wrong casing, legacy names in
   canonical directories) is not independently tested outside the real
   repository. The existing validation against 39 real directories is
   sufficient for this PR but a standalone fixture test harness should be
   considered for future robustness.
