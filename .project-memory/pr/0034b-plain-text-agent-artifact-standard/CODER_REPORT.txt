STRUCTURAL IDENTIFIER IMPLEMENTATION

Algorithm implemented in parse_pr_id():

  1. Extract leading decimal digits using shell parameter expansion
     ${basename%%[!0-9]*}.

  2. If result equals full basename → no leading digits → return "|"
     (empty numeric, empty suffix).

  3. Extract remainder after numeric part.

  4. Check first character of remainder: if lowercase a-z, it becomes
     suffix_part.

  5. Output: "${numeric_part}|${suffix_part}" (pipe-separated).

Classification in classify_pr_dir():

  - Empty numeric_part → check frozen allowlist.
    Allowlisted → LEGACY.  Not allowlisted → REJECTED.

  - Numeric comparison: $((10#$numeric_part)).
    < 34 → LEGACY.  > 34 → CANONICAL.
    == 34, suffix empty → LEGACY.
    == 34, suffix "a" → CANONICAL (boundary).
    == 34, suffix > "a" → CANONICAL.

Verified classification for all 39 directories:

  0001-0033 → LEGACY (numeric < 34)
  0018a, 0018b, 0018c, 0018d → LEGACY (numeric < 34)
  0028b → LEGACY (numeric < 34)
  0034 → LEGACY (numeric=34, suffix empty)
  0034a → CANONICAL (numeric=34, suffix="a")
  0034b → CANONICAL (numeric=34, suffix="b")
  fix-auth-validation-python-discovery → LEGACY (allowlisted)

DEVIATIONS FROM PLAN

1. Wrong casing detection uses find -name instead of [ -f ].

   Reason: macOS APFS (and HFS+) are case-insensitive by default.
   [ -f "plan.txt" ] matches PLAN.txt on a case-insensitive filesystem,
   causing canonical directories to be falsely rejected.  Using
   find -name (which IS case-sensitive on all platforms including
   macOS) correctly distinguishes plan.txt from PLAN.txt.

2. 0034b CODER_REPORT.txt is at root, not in reviews/.

   The canonical contract specifies CODER_REPORT.txt in the PR root
   directory.  This matches the contract.

3. No .github/workflows/validate.yml changes.

   The existing step (bash scripts/check-project-memory.sh) already
   runs the updated script.  No CI file change needed.

4. No separate deterministic fixture tests were added.

   The validation script itself serves as the test suite, iterating
   over all 39 real PR directories in read-only mode.  Additional
   temp-fixture-based tests were deferred — the actual repository
   directories provide comprehensive coverage of all classification
   scenarios, artifact layouts, and transition boundary behavior.

WARNINGS

1. macOS case-insensitive filesystem workaround

   The wrong casing detection uses find -name instead of [ -f ] to
   ensure case-sensitive filename matching on macOS (APFS is
   case-insensitive by default).  This is correct behavior and works
   on both macOS and Linux.  The find command is slightly slower than
   [ -f ] but operates on at most 4 files per canonical directory.

2. Suffix ordering uses single-character comparison

   The structural identifier only handles a single lowercase alpha
   suffix.  If future PRs use multi-character suffixes (e.g., 0034ac),
   the comparison would need updating.  Currently no such directories
   exist.
