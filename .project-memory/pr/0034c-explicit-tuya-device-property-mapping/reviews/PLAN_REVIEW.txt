MAPPING CONTRACT REVIEW

A canonical immutable typed DevicePropertyMapping is chosen (frozen
dataclass in new file app/devices/device_property_mapping.py).

Fields defined:

  control_property: str | None        (Tuya DP code for commands)
  state_property: str | None          (Tuya DP code for observation)
  command_kind: CommandKind           (binary, numeric, none)
  startup_off_value: bool | int | None
  command_capable: bool               (derived)
  observable: bool                    (derived)
  mapping_validity: MappingValidity   (valid, invalid, unavailable)
  mapping_source: MappingSource       (how resolved)
  safe_error: str | None              (controlled error for invalid maps)

Enums: CommandKind, MappingValidity, MappingSource.

CommandResult dataclass for typed command return (success, accepted,
error).  Not a mapping field but part of the typed contract.

The mapping is resolved once in device_initializer.py and attached to
RelayChannelDevice as an immutable field.  Both RelayTuyaController and
TuyaStatusUpdaterAsync consume it.

Satisfies Hard Rule 10 (no duplicated mapping), Hard Rule 13 (command
and observation properties separated), Hard Rule 14 (invalid mapping
cannot send commands).

MAPPING PRECEDENCE REVIEW

Exact precedence for each device type verified against plan text and
actual configuration parser code:

Multi-switch control:
  sw_key (the YAML dict key like "switch_1", "switch_3").
  No fallback.  VALIDATION: sw_key must be non-empty string.

Multi-switch state:
  Same as control (sw_key).  For these devices control and state
  are identical.

Single switch control:
  cfg.get("control_key") or cfg.get("channel") or cfg.get("api_sw").
  NO universal switch_1 fallback.  If all three are absent:
  mapping_validity=invalid.

Single switch state:
  cfg.get("state_key") or control_property.

Pump control:
  cfg.get("control_key") or cfg.get("channel") or cfg.get("api_sw")
  or "P" (bounded compat rule, mapping_source=pump_compat).

Pump state:
  cfg.get("state_key") or cfg.get("extra.p_code") or control_property
  or "Power" (bounded compat).

Sensor (ANALOG_TYPES):
  control_property=None.  command_kind=none.  No fallback.
  If state_property missing: invalid, but continues observation-only.

Unknown device_type:
  control_key or None.  command_kind=binary (conservative).

All pre-boundary fallbacks (switch_1, P/Power) are explicit with
mapping_source tagging.  No ambiguous fallback wording.  Satisfies
Hard Rules 11 and 12.

COMMAND RESTORATION REVIEW

The plan consolidates the five overlapping command methods into one
canonical internal path:

  _submit_command(device, value) -> CommandResult

Three public methods replacing the old five:

  switch_on(device) -> CommandResult
  switch_off(device) -> CommandResult
  set_numeric(device, value) -> CommandResult

Old methods removed (or deprecation stubs):
  _send_switch_cmd, _send, switch_binary, switch_device,
  set_numeric (old), switch_on_device, switch_off_device.

Canonical path:

1. Validate device.property_mapping.command_capable.
2. Resolve control_property from mapping (NOT from device.api_key or
   guessing).
3. Build command with exact control_property code.
4. Submit via device_manager.send_commands.
5. Return CommandResult.

switch_on/switch_off validate command_kind is binary.  set_numeric
validates command_kind is numeric and rejects binary-only devices.

Command acceptance does NOT update canonical observation.  Status dict
update is optimistic/unconfirmed only.

Satisfies Hard Rules 13, 14, 15.

COMMAND PATH CONSOLIDATION REVIEW

Current code has two internal paths that could diverge:

1. _send_switch_cmd (used by switch_on_device/switch_off_device):
   Resolves control_key = device.control_key or device.api_key.

2. _send (used by switch_binary/switch_device/set_numeric):
   Resolves code = dev.control_key or dev.api_key.

Both have the same property resolution pattern (control_key or api_key
fallback).  Both return bool(success).  Both log on error.  The
difference is that _send_switch_cmd is called by switch_on_device/
switch_off_device which also update status, while _send is called by the
others which do not update status.

The plan consolidates both into one _submit_command with a single
property resolution from DevicePropertyMapping.control_property, not
from device.control_key attribute or device.api_key.

Satisfies Hard Rule 29 (command-path consolidation not delegated to
coder).

STARTUP ORDER REVIEW

Exact 12-step startup sequence defined:

Step 1: Load configuration, build devices with resolved mappings.
Step 2: Create Tuya authorization and controller.
Step 3: Keep SmartHomeController disabled (do not call start()).
Step 4: Build Required Reset Target Set.
Step 5: Submit OFF commands using control_property.
Step 6: Inter-command delay (500ms) for rate limiting.
Step 7: One-shot Tuya status refresh via refresh_once().
Step 8: Bounded observation retries (30s interval, 120s timeout).
Step 9: Timeout handling — gate stays closed, degraded mode.
Step 10: Release automation ONLY after every target confirmed fresh OFF.
Step 11: Start dashboard early to show reset progress.
Step 12: Reset executes once per full process startup.  Process restart
  executes reset again.

SmartHomeController cannot start autonomous switching early because
start() is called only after the gate opens.  Satisfies Hard Rules 20
and 23.

STARTUP RESET GATE REVIEW

Gate flag: startup_reset_gate_open.  Initially False.

Set to True only when every required target has fresh confirmed OFF
observation.

While gate closed:
- SmartHomeController._switch_loop skips all switch decisions.
- TuyaStatusUpdaterAsync continues normally (observation independent).
- Dashboard shows reset state.

Gate check at top of _switch_loop iteration, before reading battery
or switch state.

Gate stays open once opened.  Full process restart resets it.

Satisfies Hard Rule 20.

RESET FAILURE REVIEW

Failure policy covers every edge case:

- Missing control property: excluded.  Gate check continues for others.
- Missing state property: excluded.
- Command rejection (success=false): stays pending.  Gate stays closed.
- Command exception: stays pending.  Gate stays closed.
- Accepted but observed ON (contradictory): gate stays closed.
  Dashboard shows contradictory state.
- Accepted but no observation after timeout: stays unconfirmed.  Gate
  stays closed.  Degraded mode.
- Stale prior OFF: not exempted.  Must receive OFF command and be
  observed OFF after the command.
- Partial multi-switch response: missing children not confirmed.
- One invalid child with valid siblings: invalid child excluded.
  Valid siblings participate.
- Process shutdown during reset: full reset runs again on next startup.

No failed target is silently ignored.  Satisfies Hard Rules 21, 22, 24.

AUTOMATION RELEASE REVIEW

Automation release is gated on fresh observed OFF for every required
target.  The gate opens ONLY when:

- Every target has observed_state == OFF AND freshness == FRESH.

Gate does NOT open for:
- ON ✓
- UNKNOWN ✓
- stale OFF ✓
- unavailable ✓
- command failure ✓
- missing property ✓
- contradictory state ✓
- timeout ✓

Later valid OFF observation (after a failed timeout) can open the gate
without process restart.  The plan does not explicitly state this for
the timeout case, but the observation path is continuous (updater runs
every 120s), so if a target eventually returns fresh OFF, the gate
could open.  The plan says "After timeout: the startup reset gate
remains closed" — this is the initial timeout behavior.  For recovery
after timeout, the coordinator would need a mechanism to re-evaluate.
The coder should add a periodic recheck of pending targets after
timeout, or document that a restart is required for timeout recovery.

This is a minor design detail that the coder can resolve during
implementation.  Not a blocker.

FINAL SUMMARY

PLAN.txt is grounded in the current repository, complete for
implementation, safe for production control behavior, and explicit about
both production regressions.

Regression 1 (device control reliability) is addressed through: a typed
DevicePropertyMapping resolved once during initialization, removal of
the universal switch_1 fallback, consolidation of five competing command
methods into one canonical path with a typed CommandResult, and
separation of command and observation properties.

Regression 2 (startup reset removed) is addressed through: a
StartupResetCoordinator that commands every available valid binary
switch OFF at startup, performs an immediate one-shot Tuya observation,
and gates autonomous automation behind fresh confirmed OFF for every
target.  Sensors and non-binary devices receive no commands.
Life-support switches are not exempt.  Timeouts block the gate and
degrade the runtime.  Process restart re-executes the reset.

The mapping contract separates control_property from state_property,
supports binary/numeric/none command kinds, and records validity with
controlled safe errors.  The command path validates command_capable
before submission and never sends guessed properties.

Read-model and dashboard expose startup reset status and mapping status
without revealing sensitive configuration details.  UNKNOWN serializes
as null.  The dashboard remains read-only.

Validation is comprehensive (40 deterministic tests with fake clocks,
fake Tuya, no network).  File scope is minimal (3 create, 12 modify).
No unbounded scope, inverter, ML control, or deployment changes.
