IMPLEMENTATION SUMMARY

Implemented the canonical device observation state for dessmonitor,
fixing the false-ON production defect caused by device_initializer.py
writing the available configuration boolean into the device status
dictionary.  Every RelayChannelDevice now initializes with
observed_state=UNKNOWN, and physical state is determined exclusively
by trusted Tuya observations with explicit timestamps and freshness
classification.

The implementation covers all 12 implementation requirements from
PLAN.txt: canonical typed observation, initialization fix, Tuya
observation algorithm, command boundary fix, automation fail-closed,
runtime/energy counter safety, read model update, dashboard update,
logging correction, ML/metrics correction, deterministic validation,
and CI integration.

CANONICAL OBSERVATION CONTRACT

DeviceObservationState (frozen dataclass):
  observed_state: ObservationValue (ON/OFF/UNKNOWN)
  observed_at: datetime | None (UTC timestamp)
  observation_source: str | None ("tuya" or None)
  Properties: is_on, is_off, is_unknown, has_observation

ObservationValue enum: ON="on", OFF="off", UNKNOWN="unknown"
ObservationFreshness enum: FRESH="fresh", STALE="stale",
  UNAVAILABLE="unavailable"

INITIALIZATION CHANGES

Every RelayChannelDevice initializes with:
  observation=make_observation_unavailable()
  status={} (empty, no available→ON fabrication)

Multi-switch: available=True stays in config; never enters status.
Single device: YAML status ignored for physical observation.
Pump: physical state starts UNKNOWN.

TUYA OBSERVATION CHANGES

Per-channel matching: state_key matched against Tuya response codes.
Value parsing: bool True/False, int 1/0, accepted strings.
Malformed values: silently skipped; prior observation preserved.
Missing parent device: observation unchanged.
Missing channel: observation unchanged.
Partial multi-switch response: only included channels updated.
Timeout/RPC exception: no observation update, no timestamp change.
tick() called after observation update.

FRESHNESS POLICY

Implemented constants:
  FRESH_MAX_AGE_SECONDS = 180.0  (3 minutes)
  STALE_MAX_AGE_SECONDS = 360.0  (6 minutes)

Classification:
  No observation (observed_at=None): UNAVAILABLE
  Age < 180s: FRESH
  180s <= age < 360s: STALE
  Age >= 360s: UNAVAILABLE
  Future timestamp: STALE (clock skew)
  Malformed observed_at: UNAVAILABLE

Clock: timezone-aware UTC, injectable via set_clock()/reset_clock().

PLAN REVIEW WARNING RESOLUTION

PLAN_REVIEW.txt identified a freshness threshold mismatch:
  PLAN.txt stated FRESH < 90s (3x default 30s interval).
  Production run.py uses TuyaStatusUpdaterAsync(interval=120).

Resolution:
  Did NOT reduce the production updater interval (remains 120s).
  Used production-aligned backend freshness constants:
    FRESH_MAX_AGE_SECONDS = 180 (1.5x the 120s polling interval)
    STALE_MAX_AGE_SECONDS = 360 (3x the polling interval)
  180s provides headroom for occasional single-cycle delays
  without going stale.  360s is the point where observations
  become unavailable and unsafe for automation decisions.

Why the 90-second plan value was not used:
  The 90s value assumed a 30s polling interval.  Production uses
  120s.  Using 90s would cause every observation to go STALE
  before the next one arrives (120s > 90s).  The 180s constant
  is aligned with the production polling interval while being
  safe for the 30s default (observations would arrive well
  within 180s in either configuration).
