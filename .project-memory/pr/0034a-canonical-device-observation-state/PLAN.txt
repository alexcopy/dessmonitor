PRODUCTION INCIDENT

After PR 0034 dashboard deployment, the following was observed:

1. Several physical relays were confirmed OFF by visual inspection and
   independent measurement.

2. The dashboard displayed those relays as ON (green "ON" badge).

3. The application's MAIN ON-device log (device_status.log) displayed
   the same false ON state.

4. Business logs contained accepted OFF commands for those relays,
   including successful Tuya send_commands responses.

5. Dashboard, logger, automation, and read model were all reading the
   same mutable RelayChannelDevice objects. The false state was
   internally consistent — the application genuinely believed the
   devices were ON.

OBJECTIVE

Create one canonical, structured, truthful device-observation model
that eliminates the false-ON production defect and prevents its
recurrence.

Key principles:
- Inventory is separated from observation.
- Physical state supports ON, OFF, and UNKNOWN.
- Observations have source and timestamp.
- Freshness is explicit and deterministic.
- Commands do not masquerade as observations.
- Dashboard, logs, automation, and state collectors consume the same
  canonical observation semantics.
- Stale or unknown observations cannot cause unsafe automation
  decisions.
- The existing false-ON production defect is covered by regression
  tests.

ROOT CAUSE

The false-ON production defect has a single root cause with three
contributing factors:

Root cause: app/device_initializer.py line 81

  status={sw_key: sw_cfg.get("available", parent_ok)}

  The multi-switch initializer writes the device's "available"
  configuration boolean directly as the value for sw_key in the
  device's status dictionary. Since available=True in devices.yaml
  means "this device channel is configured and active, not physically
  ON", this causes RelayChannelDevice.is_device_on() to interpret
  available=True as physically ON.

Contributing factor A: control_key/api_key inconsistency

  app/tuya/relay_tuya_controller.py switch_on_device() and
  switch_off_device() set status using device.api_key after a
  successful command, but multi-switch devices have api_key=None
  and control_key=sw_key. The status update writes to
  status[None] rather than status[state_key], so successful
  commands do not actually update the observed state. The stale
  available=True value from initialization persists indefinitely.

Contributing factor B: no observation timestamp

  TuyaStatusUpdaterAsync records no explicit observation timestamp
  on RelayChannelDevice. Timeout, exception, missing channel, or
  empty response leave prior values intact without any way to
  distinguish fresh data from stale data. A device that was once
  correctly observed as ON but has since been switched OFF via a
  command (whose status-update silently failed due to factor A)
  remains ON in the dashboard.

Contributing factor C: is_device_on returns only bool

  RelayChannelDevice.is_device_on() returns True or False. There is
  no third state for UNKNOWN. Every consumer of device state
  (dashboard, logs, automation, ML) must treat False as OFF even
  when no observation exists. This is fundamentally incapable of
  representing uncertainty.

CANONICAL TERMINOLOGY

INVENTORY: Static configuration required to locate and describe a
  device. Source: devices.yaml. Fields: tuya_device_id, control_key,
  state_key, load_in_wt, device_type, priority, etc. Never consulted
  to determine physical ON/OFF state.

AVAILABLE: Whether a configured device is enabled for use in the
  application. True means the device should be monitored and may be
  controlled. False means the device is disabled. Available is NOT
  physical ON/OFF state.

DESIRED STATE: The state requested by automation or an operator
  command. Stored as a command intent. Not an observation.

COMMAND RESULT: Whether submission to Tuya was accepted or failed.
  A successful send_commands response means Tuya accepted the
  command. It does NOT independently prove the physical device
  reached the requested state.

OBSERVED STATE: The last physical state successfully obtained from
  the observation source. Values: ON, OFF, UNKNOWN. Source: Tuya
  get_device_list_status response. Stored with a timestamp and
  source identifier.

OBSERVED AT: The clock time when the last successful observation
  containing the relevant channel was received. Timezone-aware
  (UTC). Null when no observation has ever been made.

OBSERVATION SOURCE: The system that produced the observation.
  Initially "tuya". Null when no observation exists.

FRESHNESS: Whether the observation is fresh, stale, or unavailable.
  Determined by comparing OBSERVED AT to the current time and
  configured thresholds. Fresh: observation age < fresh threshold.
  Stale: observation age >= stale threshold. Unavailable: no
  observation exists.

UNKNOWN: No trustworthy ON/OFF observation is currently available.
  This is a distinct value. It is not OFF. It is not ON. It is
  UNKNOWN.

The word "status" is ambiguous and must not be used alone. Every
use must specify which status: configuration status, command
status, observation status, or overall device status.

CANONICAL OBSERVATION CONTRACT

A dedicated immutable observation object is preferred over adding
more ambiguous keys to the mutable RelayChannelDevice.status dict.

A new frozen dataclass in app/devices/device_observation.py:

  @dataclass(frozen=True)
  class DeviceObservationState:
      observed_state: ObservationValue = ObservationValue.UNKNOWN
      observed_at: datetime | None = None
      observation_source: str | None = None
      freshness: ObservationFreshness = ObservationFreshness.UNAVAILABLE

  enum ObservationValue:
      ON = "on"
      OFF = "off"
      UNKNOWN = "unknown"

  enum ObservationFreshness:
      FRESH = "fresh"
      STALE = "stale"
      UNAVAILABLE = "unavailable"

  RelayChannelDevice gains a new field:
      observation: DeviceObservationState = field(
          default_factory=DeviceObservationState)

  The existing status dict and is_device_on() are preserved for
  backward compatibility during the transition period.

  is_device_on() is modified to:
      if self.observation.observed_state == ObservationValue.ON:
          return True
      if self.observation.observed_state == ObservationValue.OFF:
          return False
      return False  # UNKNOWN — log deprecation warning

  A new method is provided:
      def get_observation(self) -> DeviceObservationState:
          return self.observation

  Callers that need to distinguish UNKNOWN from OFF use:
      obs = device.get_observation()
      if obs.observed_state == ObservationValue.UNKNOWN:
          # handle unknown
      elif obs.observed_state == ObservationValue.ON:
          # handle on

  Plain boolean consumers (is_device_on) remain usable but return
  False for UNKNOWN with a deprecation warning in logs.

Every RelayChannelDevice begins with:

  observation = DeviceObservationState(
      observed_state=ObservationValue.UNKNOWN,
      observed_at=None,
      observation_source=None,
      freshness=ObservationFreshness.UNAVAILABLE
  )

The initial `status` dict is set to empty {} — NOT to
{sw_key: available}. The available=True configuration value must
never appear in the status dictionary.

   After:  status={}  # physical state begins UNKNOWN

   The cfg.get("status", {}) is loaded from YAML. Legacy status

   handled separately.

TUYA OBSERVATION UPDATE ALGORITHM

The algorithm for TuyaStatusUpdaterAsync._update_once():

1. Fetch get_device_list_status(tuya_ids) with TUYA_RPC_TIMEOUT.
2. On timeout: log warning, return. No timestamps updated.
3. On RPC exception: log error, return. No timestamps updated.
4. On empty result (no status list, no devices): log debug, return.
   No timestamps updated.
5. For each device in result["result"]:
   a. Parse each status item from the status list.
   b. For each status item, determine which channel it belongs to.
   c. For each channel:
      - Match by code (the Tuya code like "switch_1", "switch_2").
      - Parse value: bool True/False for booleans, int 1/0 for
        integers, accepted strings for string values.
      - If the channel value is a valid ON/OFF indicator, create a
        new DeviceObservationState with observed_at=utcnow(),
        observation_source="tuya", freshness=FRESH.
      - If the channel value is malformed or unparseable, log
        warning, leave the prior observation in place. Do not
        overwrite with UNKNOWN.
      - If the channel is absent from the response, leave the
        prior observation in place. Do not change observed_at.
        Do not overwrite with UNKNOWN.
   d. Call dev.tick(now_ts) only after updating observation.

Matching by parent Tuya device:
- Each device has tuya_device_id. All devices sharing the same
  tuya_device_id are grouped for a single get_device_list_status
  call.
- The response contains one status list per tuya_device_id.
- Each status item has a "code" field.
- A channel device with state_key="switch_1" matches a status
  item with code="switch_1".

Failure policy:
- A failed fetch cycle does not mark any observation stale.
  Staleness is determined by the freshness policy based on the
  last successful observed_at timestamp.
- A partial response (some channels present, some missing) updates
  only the channels that were successfully observed.
- If all channels for a device are absent from the response, the
  device's observation is not updated. No timestamp change.

FRESHNESS POLICY

Freshness is computed on the observation object, not stored as a
static value. A function compute_freshness(observation, now_utc)
returns the freshness value.

Thresholds:

  FRESH: observation exists and age < 90 seconds
         (3x the default 30-second Tuya polling interval)

  STALE: observation exists and age >= 90 seconds

  UNAVAILABLE: no observation exists (observed_at is None)

Clock: timezone-aware datetime.utcnow() (Python 3.12+).

Time zone: UTC. All timestamps are stored as UTC.

Restart behavior: after application restart, observation state is
reinitialized to UNKNOWN. No persistence of observation state
across restarts in this PR.

Future timestamps: if observed_at is in the future (clock skew),
treat as STALE (cannot be fresh from the future).

Malformed timestamps: if observed_at is not a valid datetime, treat
as UNAVAILABLE.

Test injection: freshness thresholds are configurable via
constructor parameter for deterministic testing. Default values
apply in production.

COMMAND VERSUS OBSERVATION POLICY

Command acceptance does NOT overwrite observed_state.

After a successful send_commands response:

1. A command intent record is created (in-memory, not persistent in
   this PR). Fields: device_id, requested_state (ON/OFF),
   command_id, sent_at, accepted (bool from Tuya response).

2. observed_state remains unchanged until the next successful
   observation cycle confirms or contradicts the commanded state.

3. The dashboard may optionally show "command pending" or "awaiting
   confirmation" status between command acceptance and the next
   observation. This is a separate field on the device or in the
   read model, not an override of observed_state.

4. If the observation cycle confirms the commanded state, the
   observed_state is updated normally.

5. If the observation cycle contradicts the commanded state (device
   did not reach the requested state), a warning is logged and
   observed_state reflects the observed reality.

Control key fix:
- relay_tuya_controller.py switch_on_device() and
  switch_off_device() must update status using device.control_key
  (or state_key), NOT device.api_key.
- The pattern is: send command using control_key, then if accepted,
  update the observation status using the same key.
- However, per the policy above, command acceptance alone does not
  replace observation. The in-flight status update is only for
  optimistic UI display, clearly marked as unconfirmed.
- The canonical observation update path is exclusively through
  TuyaStatusUpdaterAsync.

AUTOMATION FAIL-CLOSED POLICY

SmartHomeController._switch_loop() must fail closed for UNKNOWN or
stale observations:

1. Before reading any device state for automation decisions, check
   freshness. If freshness is STALE or UNAVAILABLE, log structured
   warning and skip the device.

2. all_devices_on() and all_devices_off() in RelayDeviceManager
   are modified:
   - all_devices_on(): returns devices where observed_state is ON.
     UNKNOWN and OFF are excluded.
   - all_devices_off(): returns devices where observed_state is
     OFF. UNKNOWN and ON are excluded.
   - These methods may also accept an optional
     "include_unknown" parameter for diagnostic use only.

3. Runtime and energy calculations (tick(), today_run_sec,
   today_wh):
   - These must not advance when observed_state is UNKNOWN.
   - tick() already checks is_device_on(). With the updated
     is_device_on() returning False for UNKNOWN, advancement is
     prevented. This is correct as long as the deprecation
     warning is not suppressed.
   - Future energy calculations (PR 0036) should use
     get_observation() to distinguish UNKNOWN from OFF.

4. Pond pump life-support: existing safety rules are preserved.
   This PR does not re-enable disabled pump automation. If pump
   automation were enabled, UNKNOWN state would prevent switching
   ON (safe) and prevent switching OFF (safe — preserves current
   state).

5. Safe skip behavior: when a device is skipped due to UNKNOWN or
   stale state, a structured log warning is emitted:
     f"[AUTOMATION] Device {name}: skipping — observation {state}
       freshness {freshness}"

6. Structured warning behavior: warnings are logged through the
   existing logging infrastructure with extra fields for Loki:
     extra={"evt": "automation_skip", "dev": name,
            "obs_state": state, "freshness": freshness}

DASHBOARD CONTRACT

The dashboard currently displays:
- "ON" badge (green) for currently_on=true
- "OFF" badge (grey) for currently_on=false

With the canonical read model:
- ON (currently_on=true, observed_state=on):
  Green badge, no change.
- OFF (currently_on=false, observed_state=off):
  Grey badge, no change.
- UNKNOWN (currently_on=null, observed_state=unknown):
  Grey badge with "?" text, or "---". Not green. Not red.
- Stale: a subtle visual indicator (border or opacity) applied
  to devices with freshness=stale. The badge text remains ON/OFF
  if the observation itself was valid when fresh.
- Unavailable device (not available in config): grey "---".

Summary counts:
- ON count: number of devices with observed_state=ON
- OFF count: number of devices with observed_state=OFF
- UNKNOWN count: number of devices with observed_state=UNKNOWN
  (new counter; never merged into OFF count)
- Total / configured count: all devices in the configured list

Dashboard index.html and dashboard.js are updated to render the
new fields. The existing polling and state machine remain
unchanged.

The dashboard does NOT add:
- write controls
- command status display (deferred)
- observation timestamps (may be added in a future PR when
  dashboard.js supports it)

Accessibility:
- UNKNOWN uses accessible text "Unknown", not colour alone.
- Stale indicator includes text explanation in a tooltip or
  adjacent text.

          "OFF" if obs.observed_state == ObservationValue.OFF else \

  freshness: "fresh" | "stale" | "unavailable"

No raw Tuya payloads, IDs, keys, or secrets are logged. The

Human-readable logs are outputs only. They must not be parsed
back into runtime state.

VALIDATION PLAN

Test structure:

  scripts/check-canonical-device-observation.sh runs Python unit
  tests with fake clocks, fake Tuya responses, and in-memory
  devices. No real Tuya calls, no network access, no real timers.

Test categories:

CANONICAL STATE (8 tests)
- Device initializes with observed_state=UNKNOWN, freshness=UNAVAILABLE
- Multi-switch available=true starts UNKNOWN, not ON
- Single switch starts UNKNOWN without trusted observation
- Analog sensor starts UNKNOWN
- Unavailable device starts UNKNOWN
- Legacy YAML status values do not override initial UNKNOWN
- DeviceObservationState is immutable (frozen)
- compute_freshness with no observation returns UNAVAILABLE

OBSERVATION UPDATE (8 tests)
- Valid Tuya true (bool) sets observed_state=ON, observed_at updated
- Valid Tuya false (bool) sets observed_state=OFF, observed_at updated
- Integer 1 sets observed_state=ON
- Integer 0 sets observed_state=OFF
- Accepted string "true"/"1"/"on" sets observed_state=ON
- Accepted string "false"/"0"/"off" sets observed_state=OFF
- Timeout does not change observed_at or observed_state
- RPC exception does not change observed_at or observed_state

MISSING CHANNEL BEHAVIOR (3 tests)
- Absent channel in response leaves prior observation unchanged
- Absent channel does not create a new UNKNOWN observation
- Partial multi-switch response updates only included channels

FRESHNESS (5 tests)
- Observation age < 90s returns FRESH
- Observation age >= 90s returns STALE
- No observation returns UNAVAILABLE
- Future timestamp returns STALE
- Freshness thresholds are configurable for test injection

COMMAND BOUNDARY (4 tests)
- Accepted OFF command does not set observed_state=OFF
- Command result is recorded separately from observation
- Observation confirmation clears pending state
- Contradictory observation is represented correctly

CONTROL KEY FIX (2 tests)
- switch_on_device uses control_key, not api_key, for status update
- switch_off_device uses control_key, not api_key, for status update

AUTOMATION SAFETY (5 tests)
- all_devices_on excludes UNKNOWN devices
- all_devices_off excludes UNKNOWN devices
- SmartHomeController skips UNKNOWN devices (does not switch)
- SmartHomeController skips STALE devices (does not switch)
- tick() does not advance counters for UNKNOWN state

API SERIALIZATION (4 tests)
- currently_on is true for ON, false for OFF, null for UNKNOWN
- observed_state serializes as "on", "off", "unknown"
- observed_at serializes as ISO-8601 string or null
- freshness serializes as "fresh", "stale", "unavailable"

DASHBOARD RENDERING (3 tests)
- UNKNOWN renders as "---" not "OFF"
- Stale row has visual indicator
- Summary count shows ON, OFF, and UNKNOWN separately

REGISTRATION (3 tests)
- No raw Tuya device IDs, control keys, or API keys in API response
- No raw Tuya device IDs, control keys, or API keys in logs
- Authentication and read-only dashboard regressions pass

RISKS AND MITIGATIONS

Risk: Consumer code still depends on is_device_on() returning bool
  without understanding UNKNOWN.
  Mitigation: is_device_on() returns False for UNKNOWN with a
  deprecation warning. All consumers are updated in this PR.
  Validation: automated tests verify all 10+ consumer paths use
  canonical observation.

Risk: RelayTuyaController status update key fix accidentally
  writes to wrong dict key.
  Mitigation: fix is minimal (change api_key to control_key in two
  lines). Unit tests verify correct key after switch_on/off.
  Validation: control key fix tests in validation plan.

Risk: TuyaStatusUpdaterAsync rewrite introduces a regression in
  observation frequency or reliability.
  Mitigation: algorithmic changes are additive (add observation
  timestamp, add per-channel matching). Existing Tuya call pattern
  is preserved. No change to RPC timeout or polling interval.
  Validation: runtime replay of recorded Tuya responses.

Risk: Dashboard shows UNKNOWN for an extended period after restart.
  Mitigation: this is correct behavior (no observation yet). The
  dashboard already handles unavailable state. Users are
  accustomed to the "connecting..." state on page load.
  Validation: production validation step 1 confirms UNKNOWN
  display.

Risk: Rollback restores the false-ON defect.
  Mitigation: documented in Rollback section. Users must
  re-deploy the fix after rollback.
  Validation: documentation and acceptance criteria.

ROLLBACK

Rollback is an application-image rollback only:

1. Revert to the previous known-good Docker image tag or digest.
2. GitOps (ArgoCD) points to the previous image.
3. No database rollback, Secret change, or config change.

Rollback risk: The previous image has the false-ON defect.
Operators deploying a rollback must be aware that the dashboard
will again show ON for devices that may be physically OFF. The
fix must be re-implemented in a subsequent deployment.

There is no data migration that needs to be reversed. No device
configuration is rewritten. No schema change requires rollback.
