WARNINGS

1. Freshness threshold mismatch with production polling interval

PLAN.txt Section FRESHNESS POLICY states the FRESH threshold as 90 seconds
(3x the default 30-second Tuya polling interval). However, run.py line 66
instantiates TuyaStatusUpdaterAsync with interval=120, which means in the
current production configuration observations arrive at most every 120
seconds. A 90-second fresh threshold would cause every observation to be
marked STALE before the next one arrives (120s > 90s). The plan says
freshness thresholds are configurable, so this is resolvable, but the
plan's explicit 90s number is inconsistent with the production polling
interval observed in run.py. The coder must reconcile this — either by
increasing the fresh threshold to accommodate 120s polling, or by noting
the discrepancy in CODER_REPORT.txt and documenting the expected
configuration.

INCIDENT AND ROOT CAUSE REVIEW

PLAN.txt correctly traces the false-ON production incident through all
affected components:

Data path traced: devices.yaml -> DeviceInitializer -> RelayChannelDevice
-> RelayDeviceManager -> status_updater_async -> web_runtime_integration
-> dashboard -> device_status_logger -> MLCollector -> TimescaleCollector

The root cause is correctly identified as device_initializer.py line 81
where sw_cfg.get("available", parent_ok) is written directly into the
status dictionary as the physical ON/OFF value for that channel.

Three contributing factors are correctly identified and distinguished from
the root cause:

Factor A - control_key/api_key inconsistency in relay_tuya_controller.py
causes successful OFF commands to write to status[None] instead of
status[state_key], allowing the stale available=True from initialization
to persist indefinitely in the state that the dashboard reads.

Factor B - No observation timestamp on any device. Timeout, exception,
empty response, or missing channel leave prior values intact without any
way to detect staleness.

Factor C - is_device_on() returns only True or False. Every consumer must
treat False as OFF even when no observation has ever been made. This is
fundamentally incapable of representing uncertainty.

The plan distinguishes confirmed facts from hypotheses requiring coder
inspection. It states "The exact devices.yaml contents, Tuya response
patterns, and restart history are not available to the planner." No
invented production facts are asserted.

The plan correctly identifies that Factor A is not the root cause but a
contributor: even if the control_key fix were applied, the initialization
would still set status to available=True, so an OFF command would be
immediately overwritten by the next observation cycle. Only with all three
fixes together (initialization, control_key, and canonical observation)
is the defect fully resolved.

CANONICAL TERMINOLOGY REVIEW

PLAN.txt defines unambiguous meanings for all required terms:

INVENTORY: Static configuration from devices.yaml. tuya_device_id,
control_key, state_key, load_in_wt, device_type, priority. Never
consulted to determine physical ON/OFF state.

AVAILABLE: Whether a configured device is enabled for monitoring and
control. True means active/monitored. NOT physical ON/OFF state. The
plan explicitly states that available remains configuration-only.

DESIRED STATE: State requested by automation or operator command. Stored
as command intent. Not an observation.

COMMAND RESULT: Whether submission to Tuya was accepted (successful
send_commands response). Does NOT independently prove physical state.

OBSERVED STATE: Last physical state successfully obtained from the
observation source. Values: ON, OFF, UNKNOWN. Stored with timestamp
and source identifier.

OBSERVED AT: Clock time of last successful observation containing the
relevant channel. Timezone-aware UTC. Null when no observation.

OBSERVATION SOURCE: System that produced the observation. Initially
"tuya". Null when no observation exists.

FRESHNESS: FRESH, STALE, or UNAVAILABLE. Determined by comparing
observed_at age to current time and configured thresholds.

UNKNOWN: No trustworthy ON/OFF observation is currently available.
A distinct value. Not OFF. Not ON.

The plan explicitly states: "The word 'status' is ambiguous and must not
be used alone. Every use must specify which status."

Available remains configuration-only. This satisfies Hard Rules 8 and 9.
No generic ambiguous status terminology is left unqualified.

OBSERVATION CONTRACT REVIEW

DeviceObservationState (frozen dataclass in
app/devices/device_observation.py):

  observed_state: ObservationValue
    ON = "on"
    OFF = "off"
    UNKNOWN = "unknown"

  observed_at: datetime | None = None

  observation_source: str | None = None

  freshness: ObservationFreshness

is_device_on() is modified to use observation:
  ON -> True, OFF -> False, UNKNOWN -> False with deprecation warning.

get_observation() provides full canonical state access.

The existing status dict is preserved for backward compatibility during
the transition. The plan explicitly prohibits removing it in this PR.

No raw Tuya payloads, device identifiers, channel keys, credentials,
tokens, or secrets are exposed. The existing _FORBIDDEN_DEVICE_ATTRS

TUYA UPDATE ALGORITHM REVIEW

PLAN.txt defines exact behavior for every edge case:

Matching parent Tuya devices: grouped by tuya_device_id for a single
get_device_list_status call.

Matching channels: by code field (state_key like "switch_1", "switch_2").

Parsing: bool True/False for booleans, int 1/0 for integers, accepted
strings for string values. Malformed -> log warning, leave prior
observation unchanged.

Missing parent device: leave prior observation unchanged. No timestamp
change.

Missing channel (absent from response): leave prior observation unchanged.
No timestamp change. Do NOT overwrite with UNKNOWN.

Partial response: update only included channels. Absent channels keep
their prior observation unchanged.

Duplicate response entries: handled naturally by per-channel matching
(last write wins, but value should be identical).

Timeout (asyncio.TimeoutError): log warning, return. No timestamps
updated.

RPC exception: log error, return. No timestamps updated.

Empty response (no result, no devices): log debug, return. No timestamps
updated.

Timestamp creation: utcnow() on each successful per-channel observation.

Atomic updates: observation updated only after successful parse.

tick() called AFTER observation update (ordering matters).

Missing data is never interpreted as OFF. Failed cycles do not advance
observed_at. This satisfies Hard Rules 10, 13, and 14.

FRESHNESS REVIEW

PLAN.txt defines:

Clock source: timezone-aware datetime.utcnow() (Python 3.12+).
Timezone: UTC. All timestamps stored as UTC.

FRESH threshold: observation age < 90 seconds.
STALE threshold: observation age >= 90 seconds.
UNAVAILABLE: no observation exists (observed_at is None).

Restart behavior: observation state reinitialized to UNKNOWN. No
persistence across restarts in this PR.

Future timestamps: treated as STALE (cannot be fresh from the future).

Malformed timestamps: treated as UNAVAILABLE.

Test injection: freshness thresholds configurable via constructor
parameter for deterministic testing. Default values apply in production.

Freshness is computed by compute_freshness(observation, now_utc), not
stored as a static value. This is deterministic.

No browser-derived time is used. Satisfies Hard Rule 12.

One warning applies (see WARNINGS section): the stated 90s threshold
assumes a 30s polling interval, but the production run.py uses a 120s
interval. The coder must reconcile this discrepancy.

COMMAND OBSERVATION BOUNDARY REVIEW

PLAN.txt defines explicit separation:

1. Accepted commands do NOT overwrite observed_state. Command intent
   records are created separately (in-memory, not persistent).

2. The dashboard may optionally show "command pending" or "awaiting
   confirmation" separate from observed_state.

3. Confirming observations (from next Tuya cycle) update observed_state
   normally. Contradictory observations log a warning and reflect actual
   observed reality.

4. Missing confirmation: if a command was accepted but the next
   observation cycle fails, observed_at does not advance. After 90s
   the observation goes STALE (or if no observation ever existed,
   UNAVAILABLE).

5. control_key fix: relay_tuya_controller.py switch_on_device() and
   switch_off_device() must update status using device.control_key (or
   state_key), NOT device.api_key. This prevents status[None] from being
   created.

6. The canonical observation update path is exclusively through
   TuyaStatusUpdaterAsync, not through command result handlers.

This satisfies Hard Rules 10 and 15. Commands cannot masquerade as
observations. UNKNOWN cannot silently become OFF or ON. The api_key=None
defect is explicitly addressed. status[None] cannot be created after the
fix.
