ROOT CAUSES

Failure 1 — check-web-authentication.sh test [32] (throttling HTTP):
The throttling test obtained a CSRF token via a urllib GET (which creates its
own cookie jar), then attempted to POST with `post_raw()` using a separate empty
cookie jar (cj9).  The CSRF token was associated with the GET session cookie, but
the POST sent no matching session cookie.  The server correctly rejected every
attempt at CSRF validation (403), preventing the requests from reaching the
throttling codepath.  Fix: replaced the `get()` + `post_raw()` split with a single
urllib-based opener that shares the cookie jar between the CSRF-token-retrieval
GET and all throttling POSTs, so the session cookie is carried through.

Failure 2 — check-web-control-state-provider.sh tests [27] and [28]:
These tests called `create_app()` without any WEB_AUTH_* environment variables.
After PR 0033, `create_app()` is fail-closed — it raises `WebAuthConfigError`
when the operator username is missing.  Fix: each test now generates a fresh
Argon2 hash and 128-hex-char session secret at runtime via `argon2.PasswordHasher`
and `secrets.token_hex(64)`, sets both plus `WEB_AUTH_USERNAME` and
`WEB_AUTH_TEST_HTTP` on `os.environ` before calling `create_app()`.  No fixed
credentials appear in the repository.

Failure 3 — check-runtime-read-only-web-host.sh test [19] (embedded server HTTP):
`start_runtime_read_only_web_host()` calls `create_app()` without forwarding
its own `environ` parameter.  Fix: the test now sets WEB_AUTH_* variables on
`os.environ` before calling `start_runtime_read_only_web_host()`, and cleans
them up afterwards.  The test also now logs in (GET /login → extract CSRF →
POST /login with correct credentials) before fetching /control/state, matching
the authenticated contract.
