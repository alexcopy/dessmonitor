# Autonomous Control and Operator Override

This document records the architectural decision and design principles for
autonomous control vs. manual operator override in the dessmonitor control system.

## 1. Autonomous Operation Is the Default Path

The system's default operating mode is **autonomous**. The policy decision engine
(`evaluate_policy_decision` in PR 0018C) evaluates battery voltage, energy budget,
pond safety, weather forecast, and schedule profiles to produce advisory decisions.
Command intent arbitration (PR 0020) translates those decisions into command intents
and proposals **without requiring human approval for every command**.

Autonomous proposals are `execution_eligible=true` by default when not blocked.
Future safety gates remain the final authority before any future execution.

## 2. Operator/Web UI Input Is Override/Correction, Not Required Approval

Manual operator input is an **override and correction layer**, not a mandatory
approval gate. The manual control queue (PR 0019) allows operators to enqueue
intents, but those intents are only consumed when they exist — the system does
not block waiting for human input when the queue is empty.

When both autonomous policy and manual queue items are present, the manual queue
takes priority (operator override). This ensures the operator can always correct
or countermand an autonomous decision when needed.

## 3. Future Web UI Writes Operator Intent Into Control Layer

Future web UI components will write operator intent into the control layer through
the manual control queue. The web UI will **not** directly call Tuya, hardware
adapters, or any runtime service. All operator actions will flow through:

```
Web UI → Manual Control Queue → Command Arbitration → Command Proposal → (future) Safety Gates → (future) Executor
```

## 4. Future Web UI Must Not Directly Execute Tuya/Hardware

Under no circumstances will a web UI component directly:
- Call `switch_on_device` or `switch_off_device`
- Call any Tuya API method
- Call any hardware adapter or relay controller
- Bypass the command arbitration or safety gate layers

The control layer is the single source of truth for all command execution.

## 5. Safety Gates Remain Final Authority

Safety gates are the final authority before any command proposal is executed.
Even after a proposal is generated by command arbitration:
- Safety gates may reject the proposal
- Safety gates may modify execution parameters
- Safety gates may delay execution pending condition checks
- Safety gates enforce hardware-level invariants (max load cap, cooldown, health)

Safety gates are deferred to a future PR (post-0020).

## 6. Life-Support Constraints Remain Above Discretionary Convenience

Pond/fish aeration is life-support, not a discretionary convenience. The policy
decision engine (PR 0018C) enforces this invariant. Command arbitration does not
override life-support decisions — it simply translates them into proposals.

The operator may override an aeration decision, but future safety gates retain
authority to reject that override if it would endanger life-support.

## 7. Manual Override May Be Rejected by Future Safety Gates

Even when an operator enqueues a manual command, future safety gates may:
- Reject the override if it violates hardware safety limits
- Reject the override if it would exceed inverter max load
- Reject the override if it conflicts with life-support protection
- Flag the override for operator review before executing

The operator's override is advisory to the safety layer, not a bypass.

## 8. Future Controlled Execution Remains Separate

Command proposals are generated by arbitration but execution is deferred.
Future PRs will implement:
- Safety gate evaluation (pre-execution checks)
- Controlled execution eligibility (time windows, cooldown, load cap)
- Executor dispatching (Tuya adapter integration)
- Execution logging and audit trail

## 9. ML Advisory Remains Later

ML models may, in the future, provide advisory recommendations for:
- Voltage threshold tuning
- Weather forecast confidence curves
- Load classification suggestions
- Schedule window optimization

ML advisory output will be fed into the policy decision engine as input, not as
direct control. The policy decision engine retains final authority over all decisions.
ML advisory is deferred to a future PR.

## 10. ML Control Remains Much Later

Machine learning models must **never** directly control devices without separate
safety-reviewed approval per ADR-0003. ML control is a distinct milestone from
ML advisory. Before any ML model can directly influence relay states or bypass
the safety gate layer, it must pass:
- Independent safety review
- Shadow/advisory mode validation
- Controlled rollout with kill-switch capability
- Explicit operator opt-in

ML control is deferred to a much later milestone and is NOT in scope for any
current or near-term PR.
